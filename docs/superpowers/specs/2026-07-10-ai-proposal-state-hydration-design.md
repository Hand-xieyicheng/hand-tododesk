# AI 提案状态刷新设计

## 背景

AI 提案创建时，完整提案会作为快照写入 `AiMessage.metadataJson`。确认执行后，真实状态和操作结果写入 `AiActionProposal` 与 `AiActionItem`，但消息快照不会同步更新。

当前会话通过前端内存替换立即展示最新状态；切换会话后，消息列表重新读取旧快照，已成功执行的提案再次显示为 `PENDING_CONFIRMATION`，并重新出现“取消提案”“保存修改”“确认执行”按钮。再次点击确认时，后端根据提案表中的真实终态拒绝执行并返回 `AI proposal cannot be executed`。

## 目标

- `AiActionProposal` 和 `AiActionItem` 是提案状态及执行结果的唯一可信来源。
- 每次加载会话消息时，提案消息返回当前真实提案，而不是创建时的旧快照。
- 已执行成功的提案继续显示为只读结果卡，展示“执行成功”和每项结果，不显示待确认操作按钮。
- 同一读取规则同时覆盖取消、过期、执行失败、部分失败和重试后的状态。

## 方案

在 `AiStore.listMessages` 完成消息分页和反序列化后，对本页 `kind === "PROPOSAL"` 且包含提案标识的消息进行状态刷新：

1. 从消息元数据中读取提案 ID。
2. 通过现有 `getProposal(userId, proposalId)` 读取当前 `AiActionProposal` 及其 `AiActionItem`。
3. 找到提案时，用最新提案替换返回消息中的 `metadata.proposal`。
4. 提案不存在时保留原消息元数据，避免历史消息列表因单条异常而整体不可用。

本次不修改消息表中的历史快照。消息快照仍可保留创建时数据，但 API 返回值在读取边界上由真实提案状态覆盖。

## 数据流

确认执行后的数据流为：

1. 执行器更新 `AiActionProposal.status` 和各 `AiActionItem` 结果。
2. 当前前端继续使用确认接口返回的最新提案立即更新卡片。
3. 用户切换会话后，`GET /ai/sessions/:id/messages` 调用 `listMessages`。
4. `listMessages` 用提案表当前数据刷新提案消息元数据。
5. `AiProposalCard` 收到 `SUCCEEDED` 等终态，现有状态判断自动隐藏编辑和确认按钮。

## 备选方案及取舍

- 在每个状态变更路径同步更新 `AiMessage.metadataJson`：读取更直接，但确认、取消、过期、失败、部分失败和重试路径都必须维护双写，后续容易再次遗漏。
- 前端为每张提案卡单独请求详情：需要新增接口并产生额外请求，状态一致性责任也会扩散到客户端。

选择读取时刷新，因为它把一致性规则集中在消息存储边界，并保留提案表作为单一状态源。

## 错误处理

- 消息没有合法提案 ID：按普通历史快照返回。
- 对应提案已经不存在：保留原元数据，不阻断整个会话加载。
- 数据库读取失败：沿用现有消息列表错误处理，返回请求错误，不伪造提案状态。

## 测试

- 后端存储回归：消息元数据为 `PENDING_CONFIRMATION`，提案表为 `SUCCEEDED`，`listMessages` 必须返回 `SUCCEEDED` 及最新操作结果。
- 现有分页顺序和 `nextCursor` 行为保持不变。
- 前端组件验证：`SUCCEEDED` 卡片保留结果展示，不出现“取消提案”“保存修改”“确认执行”。
- 运行 API 与桌面端相关测试、类型检查和差异格式检查。

## 非目标

- 不修改提案执行、幂等或重试语义。
- 不删除历史提案卡。
- 不新增提案详情接口。
- 不迁移或批量重写现有 `AiMessage.metadataJson`。
